# CozyBooking — backend системы аренды жилья

Полнофункциональное back-end-приложение (REST API) для аренды жилья: объявления,
поиск и фильтрация, роли пользователей, бронирования и отзывы. Финальный проект
курса Python Advanced (ICH).

**Стек:** Django 5 · Django REST Framework · JWT (SimpleJWT) · MySQL (осн.) / SQLite (dev) · Docker.

---

## 1. Архитектура

Проект построен по **слоёной (layered) архитектуре** — каждый слой отвечает строго
за своё. Это главное, что оценивается по критериям «Архитектура» и «Качество кода».

```
cozybooking/
├── config/                 # настройки проекта (settings, главный urls, wsgi/asgi)
├── apps/                   # ВСЕ приложения — отдельным пакетом
│   ├── urls.py             # роутер, собирающий urls всех приложений
│   ├── common/             # общий слой: базовая модель, permissions, ошибки, хелперы
│   ├── users/              # пользователи, роли, регистрация/логин/логаут (JWT)
│   ├── listings/           # объявления, поиск/фильтры/сортировка, история поиска/просмотров
│   ├── reservations/       # бронирования, пересечение дат, переходы статусов
│   └── reviews/            # отзывы и рейтинги
├── manage.py
├── requirements.txt
├── Dockerfile / docker-compose.yml / entrypoint.sh
└── .env.example
```

Внутри каждого приложения — одинаковые слои:

| Папка | Ответственность | Правило |
|---|---|---|
| `choices/` | Перечисления (роли, статусы, типы жилья) | только константы-enum |
| `constants/` | Статические значения (ключи фильтров, размеры страниц) | только константы |
| `dto/` | Сериализаторы | **только** валидация формы и (де)сериализация |
| `errors/` | Доменные классы ошибок | понятный код + HTTP-статус |
| `filters/` | Поиск, фильтрация, сортировка | чистая работа с ORM |
| `paginations/` | Пагинация | срез + метаданные |
| `models/` | Модели БД | описание таблиц |
| `repositories/` | Запросы в БД | **ТОЛЬКО** запросы, ничего лишнего |
| `services/` | Бизнес-логика | «как правильно»: правила, расчёты, проверки прав |
| `controller/` | Views (обработчики маршрутов) | принять запрос → валидировать → отдать сервису |
| `urls.py` | Маршруты приложения | только свои эндпоинты |

**Поток запроса:**

```
HTTP → controller → (dto: валидация) → service (бизнес-правила) → repository (БД) → ответ
```

Почему так: контроллер не знает про SQL, репозиторий не знает про HTTP, бизнес-правила
лежат в одном месте и легко тестируются. Каждый слой можно менять независимо.

## 2. Быстрый старт (локально, SQLite)

Для локальной разработки БД по умолчанию — SQLite (не нужен MySQL-сервер).

```bash
# 1. Виртуальное окружение
python -m venv .venv
source .venv/Scripts/activate      # Windows (Git Bash)
# source .venv/bin/activate        # Linux/macOS

# 2. Зависимости
#    Локально без MySQL можно ставить без mysqlclient (drf-spectacular нужен —
#    он в INSTALLED_APPS, без него проект не стартует):
pip install Django==5.1.4 djangorestframework==3.15.2 djangorestframework-simplejwt==5.3.1 django-environ==0.11.2 drf-spectacular==0.27.2 gunicorn==23.0.0
#    (полный requirements.txt с mysqlclient используется в Docker)

# 3. Файл окружения
cp .env.example .env               # MYSQL=False → SQLite

# 4. Миграции и запуск
python manage.py migrate
python manage.py createsuperuser   # по желанию, для /admin
python manage.py runserver
```

API поднимется на `http://127.0.0.1:8000/`.

## 3. Запуск через Docker (MySQL)

Основная БД по требованиям проекта — MySQL. Весь стек поднимается одной командой:

```bash
cp .env.example .env               # значения БД для контейнера
docker compose up --build
```

Что происходит: поднимается контейнер `db` (MySQL 8), затем `web` ждёт готовности БД
(`entrypoint.sh`), применяет миграции, собирает статику и стартует через **gunicorn**.
Compose сам выставляет `MYSQL=True` и `DB_HOST=db` для веб-контейнера.

## 5. API — эндпоинты

Базовый префикс: `/api/`. Аутентификация — JWT: заголовок `Authorization: Bearer <access>`.

**Как смотреть и тестировать API (без фронтенда — это backend-проект):**
- **Swagger UI:** `http://127.0.0.1:8000/api/docs/` — интерактивная документация: все
  эндпоинты, схемы полей, кнопка «Try it out» (авторизация по JWT через «Authorize»).
- **Django Admin:** `http://127.0.0.1:8000/admin/` — веб-панель для данных (нужен `createsuperuser`).
- **Postman:** импортируй `docs/CozyBooking.postman_collection.json` — готовый сценарий
  из 18 запросов (логины сами сохраняют токены и id в переменные).

### Пользователи (`/api/users/`)
| Метод | Путь | Доступ | Описание |
|---|---|---|---|
| POST | `/register/` | все | регистрация (`name`, `email`, `password`, `role`) |
| POST | `/login/` | все | вход → `access`, `refresh`, данные пользователя |
| POST | `/logout/` | авторизован | выход (гасит `refresh` через blacklist) |
| POST | `/token/refresh/` | все | обновление `access` по `refresh` |

### Объявления (`/api/listings/`)
| Метод | Путь | Доступ | Описание |
|---|---|---|---|
| GET | `/` | все | каталог с фильтрами (см. ниже) |
| POST | `/` | арендодатель | создать объявление |
| GET | `/<id>/` | все | карточка (фиксирует просмотр) |
| PATCH | `/<id>/` | владелец | редактировать |
| DELETE | `/<id>/` | владелец | удалить |
| POST | `/<id>/availability/` | владелец | вкл/выкл видимость |
| GET | `/my/` | арендодатель | мои объявления (в т.ч. скрытые) |
| GET | `/popular-searches/` | все | популярные поисковые запросы |

**Query-параметры каталога:** `search`, `location`, `price_min`, `price_max`,
`rooms_min`, `rooms_max`, `housing_type`, `order` (`price`/`-price`/`created_at`/`-created_at`/`popular`/`-popular`), `page`, `page_size`.

### Бронирования (`/api/reservations/`)
| Метод | Путь | Доступ | Описание |
|---|---|---|---|
| POST | `/` | арендатор | создать бронь (`listing`, `start_date`, `end_date`) |
| GET | `/my/` | авторизован | мои брони |
| GET | `/incoming/` | арендодатель | брони по моим объявлениям |
| PATCH | `/<id>/status/` | стороны брони | сменить статус |

### Отзывы (`/api/reviews/`)
| Метод | Путь | Доступ | Описание |
|---|---|---|---|
| POST | `/` | арендатор | отзыв по своей завершённой броне |
| GET | `/listing/<id>/` | все | все отзывы объявления |

## 6. Роли и бизнес-правила

**Роли:** `RENTER` (арендатор) и `LESSOR` (арендодатель). Разграничение — через
permission-классы (`apps/common/permissions.py`) и проверку владения в сервисах.

**Бронирование — статусы:** `PENDING → CONFIRMED/REJECTED → CHECKED_IN`, либо `CANCELED`.
- Пересечение дат: два интервала заняты, если `start_A < end_B AND end_A > start_B`
  (отменённые/отклонённые брони не считаются занятостью).
- Даты не в прошлом, `start_date < end_date`, нельзя бронировать своё объявление.
- Арендатор может только **отменить** и не позднее чем за 2 дня до заезда.
- Арендодатель может **подтвердить/отклонить/отметить заселение**.

**Отзыв** можно оставить только по своей броне со статусом `CHECKED_IN` и только один раз.

## 8. Безопасность

Что заложено в проекте (пригодится и на защите):

- **Целостность бронирований:** явная стейт-машина статусов
  (`ALLOWED_TRANSITIONS`) — нельзя воскресить отменённую бронь или откатить
  заселение; создание брони под `SELECT ... FOR UPDATE` — нет гонки двойного
  бронирования.
- **Пароли:** прогон через `AUTH_PASSWORD_VALIDATORS` (длина, общеизвестные,
  «только цифры»), хеширование `set_password`.
- **Троттлинг:** анонимы/пользователи ограничены по умолчанию, auth-эндпоинты
  (`login`/`register`/`logout`/`refresh`) — жёсткий scope `5/min` против брутфорса.
  *Примечание: при нескольких воркерах gunicorn для точного лимита нужен общий
  кэш (Redis) — сейчас `LocMemCache` считает по воркеру.*
- **Fail-closed конфиг:** в проде отсутствие `SECRET_KEY` роняет старт
  (не поднимается с известным дефолтом).
- **Прод-заголовки:** вне `DEBUG` включаются HTTPS-redirect, HSTS, secure-cookies,
  `SECURE_PROXY_SSL_HEADER` (за gunicorn+прокси).
- **Доступ по ролям + владению:** permission-классы + проверки в сервисах;
  забронировать скрытое объявление нельзя; отзыв — только по своей `CHECKED_IN`-броне
  (плюс `OneToOne` на уровне БД от дублей).
- **Swagger** вне `DEBUG` доступен только администратору.

## 9. Шпаргалка к защите

> На защите вопросы преподавателя весят ×3 — понимание своего кода важнее всего.
> Ниже — вероятные вопросы и короткие ответы «своими словами».

**— Почему приложения в пакете `apps/`, а не в корне?**
Чтобы корень оставался чистым, а прикладной код был сгруппирован и подключался
одним роутером `apps/urls.py`. Плюс имена приложений не конфликтуют с библиотеками.

**— Зачем разделять `repository`, `service`, `controller`?**
Разделение ответственности. Контроллер — только HTTP (принять/провалидировать/ответить).
Сервис — бизнес-правила («как правильно»). Репозиторий — только запросы в БД.
Так код читаемый, тестируемый и меняется по частям без «спагетти».

**— Где лежит бизнес-логика и почему не в сериализаторе?**
В сервисах. Сериализатор проверяет только форму данных (типы, обязательность).
Правила (пересечение дат, права на смену статуса, кто может оставить отзыв) — в сервисе,
потому что их надо переиспользовать и тестировать независимо от HTTP-слоя.

**— Как работает проверка пересечения дат?**
Два интервала пересекаются, если `start_A < end_B` И `end_A > start_B`. Запросом в БД
ищем брони на то же объявление с таким пересечением, исключая отменённые/отклонённые.
Если нашлась хоть одна — бронировать нельзя (`apps/reservations/services/reservation_service.py`).

**— Как реализованы роли и защита маршрутов?**
Кастомная модель `User` с полем `role`. Permission-классы `IsLessor`/`IsRenter`
проверяют роль на входе во view; владение конкретным объектом дополнительно проверяет
сервис (например, редактировать объявление может только его владелец).

**— Как хранится пароль?**
Через `set_password` (хеш PBKDF2), в БД plain-text не попадает. Логин по email.
JWT: access живёт 60 мин, refresh 7 дней, при logout refresh уходит в blacklist.

**— Как устроены поиск/фильтрация/сортировка?**
Класс `ListingFilter` навешивает на queryset условия: поиск по заголовку+описанию
(`icontains`), фильтры по цене/комнатам/типу/локации, сортировка по цене/дате/популярности.
Популярность — `annotate(Count('views'))`. Пагинация — отдельный `Paginator` (срез + total).

**— Что за «дополнительный функционал»?**
История просмотров (`ViewHistory`) и история поиска (`SearchHistory`) — каждая запись
пишется при обращении; популярные объявления сортируются по числу просмотров,
популярные запросы — по частоте (`/api/listings/popular-searches/`).

**— Почему единый обработчик ошибок?**
`custom_exception_handler` приводит все ошибки к формату `{"error": {"code", "detail"}}` —
клиент всегда получает предсказуемый ответ, а не где-то строку, где-то список.

**— Как переключается БД SQLite ↔ MySQL?**
Флагом `MYSQL` в `.env`. Локально удобно на SQLite, основная БД — MySQL; в Docker
compose выставляет `MYSQL=True` и `DB_HOST=db` автоматически.

**— Как задеплоить?**
`docker compose up --build`: контейнер MySQL + контейнер приложения (gunicorn).
`entrypoint.sh` ждёт БД, применяет миграции, собирает статику. Для AWS — тот же образ
на EC2 с внешним `.env`.
